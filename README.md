# ADB Pipeline Orchestrator & Grand Database Merger

This repository contains the data processing pipelines for analyzing and merging real estate data from transaction records (**ADB1**) and RERA records (**ADB2**) into a unified **Grand Database**.

## 🏗 Architecture Overview

The system is split into three core processing engines, seamlessly connected by a unified Orchestrator:

1. **ADB1 (`ADB1/run.py`)**: Processes raw property transaction records, aggregating data at City, Location, and Project levels to generate Overall, YoY (Year-over-Year), and QoQ (Quarter-over-Quarter) market metrics.
2. **ADB2 (`ADB2/master.py`)**: Processes raw RERA (Real Estate Regulatory Authority) data, analyzing project phases, tower completion dates, sold/unsold unit metrics, and FSI. Outputs detailed RERA summaries.
3. **Grand Database (`Grand_DataBase.py`)**: The combination layer. It takes the output tables generated by ADB1 and ADB2, merges them logically using specific geographic and entity IDs (`city_id`, `loc_id`, `proj_id`), and coalesces overlapping columns to create the ultimate unified view.
4. **Orchestrator (`Orchestrator.py`)**: The master CLI script that manages everything. It safely passes environment parameters down to the pipelines, runs ADB1 and ADB2 in parallel utilizing multiple CPU cores, and subsequently triggers the Grand Database merge.

## ⚙️ Features

* **Parallel Execution**: ADB1 and ADB2 run simultaneously when executing the "All Pipelines" option.
* **Smart Upsert Logic**: You can select specific cities to update. If you choose "Replace" (Upsert) mode, the pipelines will only drop and overwrite data pertaining to your selected cities in the final Database and Excel files, leaving other records untouched.
* **Database & Excel Saves**: Data is seamlessly exported to local `.xlsx` files inside the `Output/` directory and written to the Postgres Database if configured.

## 📋 Prerequisites

* **Python 3.9+**
* **PostgreSQL Database** running and accessible.
* **`.env` file** in the root directory (see configuration below).

### Environment Configuration (`.env`)

Create a `.env` file in the root directory (`d:\Adb1_code\DB_ADB\.env`) to configure your database connection:

```env
DB_HOST=localhost
DB_PORT=5432
DB_NAME=DB-ADB
DB_USER=postgres
DB_PASSWORD=your_password
SAVE_TO_DB=true
```

## 🚀 Usage

The easiest and safest way to interact with the pipelines is by running the interactive Orchestrator:

```bash
python Orchestrator.py
```

### Execution Steps
When you run the Orchestrator, you will be prompted to configure your run:

1. **Choose Update Mode**
   * **REPLACE**: (Upsert Mode) Useful when you only want to process a few specific cities and *add/update* them in the master output files without deleting other previously processed cities.
   * **FRESH START**: Completely wipes existing Output Excel files and Database tables, processing the selected cities from scratch.
2. **City Selection**
   * The Orchestrator automatically queries the raw database for all unique cities. You can type a comma-separated list of numbers (e.g., `1, 3`) or press `Enter` to run for ALL cities.
3. **Pipeline Selection**
   * Choose to run ADB1 exclusively, ADB2 exclusively, run the final Grand Database merge, or run **ALL** (which executes ADB1+ADB2 in parallel, followed by Grand DB).
